# Debugging

当用户遇到编译错误、运行错误、崩溃、烧录失败、串口无输出、屏幕不显示、功能异常时使用本技能。

## 核心原则

- 先复现，再修复。
- 保留原始错误信息，不凭印象猜。
- 一次只改变一个主要变量。
- 从最小可验证路径开始，不做大范围重构。
- 修复后给出复现前后对比和验证步骤。

## 信息收集

先尽量确认：

- 完整错误日志或异常信息。
- 复现步骤和预期行为。
- 最近改动了哪些文件或配置。
- 操作系统、编译器、SDK、依赖版本。
- 对嵌入式项目：芯片型号、开发板、调试器、供电、下载方式、串口号、波特率、屏幕型号。

如果信息不足但可以从仓库中继续调查，先做只读检查。

## 调试流程

1. 定位失败阶段：编译、链接、启动、运行、外设初始化、UI 刷新、通信。
2. 找最小复现：最短命令、最小输入、最小硬件路径。
3. 根据证据提出 1-3 个可能根因。
4. 优先验证最可能且代价最低的根因。
5. 做最小修复。
6. 运行构建、测试或手工验证步骤。

## 常见软件问题

- 编译错误：检查包含路径、宏、类型定义、函数签名、条件编译。
- 链接错误：检查符号定义、库顺序、链接脚本、弱符号覆盖。
- 运行崩溃：检查空指针、越界、生命周期、并发、栈溢出。
- 行为不符：检查输入数据、状态机、缓存、时序、错误分支。

## 嵌入式常见问题

- 烧录失败：检查供电、连接、BOOT 引脚、读保护、下载算法、目标电压、链接地址。
- 串口无输出：检查波特率、TX/RX 引脚复用、时钟、重定向、DMA/中断、是否卡在 HardFault。
- 屏幕不显示：检查背光、复位脚、初始化序列、接口时序、显存地址、颜色格式、LVGL flush。
- 触摸异常：检查 I2C/SPI、复位/中断脚、坐标旋转、校准、驱动任务。
- FreeRTOS 异常：检查任务栈、优先级、阻塞调用、ISR API、heap 配置。

## 输出要求

回复中说明：

- 根因或当前最可能根因。
- 改了什么，为什么这样改。
- 已运行的验证命令或硬件验证步骤。
- 如果无法本机验证硬件，列出用户在板子上应执行的具体检查。

# Simple Code

当用户要求实现功能、补一个接口、加一段逻辑、修一个小问题时使用本技能。

## 目标

用最小、清楚、可验证的代码完成当前需求。不要提前设计未来，不要顺手重构。

## 实现前

先确认：

- 用户真正要求的行为是什么。
- 哪些文件和调用路径必须改。
- 是否有现有函数、工具、风格可以复用。
- 如何验证功能已完成。

如果需求含糊，先说明假设；风险高时先问。

## 实现原则

- 优先改现有最接近的位置。
- 优先复用现有模式。
- 不新增依赖，除非必要。
- 不为单一用途创建复杂抽象。
- 不添加未要求的配置、缓存、扩展机制、策略模式、监控或后台任务。
- 不批量格式化文件。
- 不改变无关 API、文件结构或命名。

## 代码形态

好选择：

- 一个小函数。
- 一个小条件分支。
- 一个明确的错误返回。
- 一个贴近现有测试风格的测试。

谨慎选择：

- 新类层次。
- 通用框架。
- 大量宏或元编程。
- 多文件重排。
- 同时改调用协议和存储格式。

## 测试和验证

根据项目选择最直接验证：

- 单元测试。
- 集成测试。
- 构建命令。
- lint/typecheck。
- 手工步骤。
- 嵌入式项目中的编译、烧录、串口输出、屏幕显示或外设现象。

没有测试条件时，说明未验证原因，并给出用户可执行步骤。

## 完成回复

简洁说明：

- 改了哪些文件。
- 新行为是什么。
- 为什么没有做更复杂方案。
- 验证命令和结果。

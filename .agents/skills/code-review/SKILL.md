# Code Review

当用户要求“检查代码”“审查代码”“找 bug”“看看有没有风险”“review 这个改动”时使用本技能。

## 审查目标

- 找真实 bug，而不是表达个人风格偏好。
- 优先关注行为错误、回归风险、边界条件、安全问题、资源泄漏和缺失测试。
- 只评论与本次变更或用户指定范围有关的问题。
- 不要求无关重构，不建议大范围改写。

## 工作流程

1. 先确认审查范围：当前 diff、指定文件、指定函数，还是整个项目。
2. 阅读相关调用路径和测试，不只看单个片段。
3. 按严重程度排序问题。
4. 每个问题都给出文件、行号、触发条件和影响。
5. 如果能给出最小修复方向，简短说明；不要把审查变成完整重写。

## 输出格式

优先输出 Findings：

```text
Findings
- [P1] file:line 简短标题
  说明触发条件、实际影响、为什么这是问题。
```

严重度参考：

- `P0`：会导致数据损坏、安全事故、无法启动或大面积不可用。
- `P1`：常见路径会失败，或明显回归。
- `P2`：边界条件、兼容性、错误处理、测试缺口。
- `P3`：低风险问题或可维护性提醒。

如果没有发现问题，明确说：

```text
没有发现明确 bug。剩余风险：未运行硬件验证/未覆盖某某路径。
```

## 审查重点

- 输入为空、越界、重复、并发、超时、失败返回。
- API 契约是否被破坏。
- 错误是否被吞掉或错误码含义改变。
- 资源是否释放：文件、内存、锁、句柄、DMA、任务、队列。
- 测试是否覆盖新增行为和失败路径。
- 修改是否超出用户要求。

## 嵌入式代码特别关注

- 中断上下文是否调用了不可在 ISR 中使用的 API。
- FreeRTOS 任务栈、优先级、阻塞时间和锁使用是否合理。
- DMA buffer、cache clean/invalidate、内存对齐是否正确。
- `volatile`、寄存器访问、启动文件、链接脚本、时钟配置是否被误改。
- LVGL flush、tick、buffer 生命周期、颜色格式和线程上下文是否正确。
